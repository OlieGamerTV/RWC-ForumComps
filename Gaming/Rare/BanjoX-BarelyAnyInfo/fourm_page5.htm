I received the XBE executables (thanks Skill!) and quickly skirmished through them, as I don't have much free time lately. I focused on the XD_Xbox one, as the largest files mostly have the best goodies inside.

Most of the info is probably going to be old/known news for the admins or anyone who played with the XBEs, but I'll post it anyway for the rest of the forum mates. If there's any info you'd not want to be leaked - I'm sorry, feel free to delete this post. Here we go!

"BanjoXD_Xbox.xbe" time stamp is 14:12:50 2004-06-24
It was built with standard Visual Studio settings, as the string "This program cannot be run in DOS mode." can be found inside the executable. :)

The following libraries were included:

XACTENG
D3DX
XGRPH
DSOUND
WMADEC
XONLINE
XNET
BINK/BINK32/BINK32A/BINK16/
BINK4444/BINK5551/.BINKYUY2/
BINK16MX/BINK16X2/BINK16M/
BINK32MX/BINK32X2/BINK32M.XPP
(probably testing different movie compression types?)

and a Dolby Surround support library.The build's directory was D:\ClientRoot\BanjoX\game\XBox_Debug\BanjoXD_Xbox. exe

There's also a nice "TESTTESTTESTTESTTESTTESTTESTTEST" string in the image's header, indicating that it was indeed a test build.

There's a very nice gap, filled with CC values starting from offset 0x0001E020 to 0x0003920F - and that's probably the starting point for storing the assets, as the later data contains variables as actorHitMsg, dyingMsg, initialRot (rotation), initialPos (position), deflectionDelta, localDeflectionDelta, wantedPos or myCylind.
The textures are probably stored at the end, as there are variables such as aid_texture_test_flash or aid_texture_test_sav.

And now for the fun part! Starting from 0x0055ADBC there's a very nice chunk of uncompressed lines of code as well as error codes. Why it is fun? Because the code is using *.C files from location d:\clientroot\ghoulies\game\ indicating that it actually used code from the Grabbed By The Ghoulies game, which was released a year ago. Some of the *.C files used are

d:\clientroot\ghoulies\game\actor\actor_body\actor _body_attack\actor_body_attack_select\actor_body_a ttack_select_debug.c

d:\clientroot\ghoulies\game\actor\actor_body\actor _body_attack\actor_body_attack_select.c

d:\clientroot\ghoulies\game\actor\actor_body\actor _body_anim\actor_body_anim_event\actor_body_anim_e vent_weapon.c

d:\clientroot\ghoulies\game\actor\actor_body\actor _body_attach\actor_body_attach_target.c

d:\clientroot\ghoulies\game\actor\actor_body\actor _body_attack\actor_body_attack_damage.c
As well as some Xbox LIVE code!

d:\clientroot\ghoulies\game\live\live_main.c

d:\clientroot\ghoulies\game\live\live_friends\live _friends_list.c

d:\clientroot\ghoulies\game\live\live_main\live_ma in_debug.c
There are also various types listed there, for example:

TYPES list from d:\clientroot\ghoulies\game\actor\actor_body\actor _body_attack\actor_body_attack_reaction.c

Reaction_Furnace, Reaction_GrimReaper, Reaction_DeathCurse, Reaction_PopCans, Reaction_WaterPistol, Reaction_FireExtinguisher, Reaction_BlunderbussKnockBack, Reaction_Bandage, Reaction_Shocked, Reaction_Candle, Reaction_MultipleThrownWeapon, Reaction_TemporaryWeapon
In other words - Banjo X probably was planned to be released, using the same engine as Ghoulies, or at least most parts of it. When I'll have some more time I'll try to actually check whether you could extract any valid asset data from the XBE.

Double post, wooo. Probably more old news for the admins.

Alright, from what I saw, using several tools, only the debug release has the weird certificate issue, other ones seems to be fine. And the XBEs are actually bootable, however every single one of them causes a very nice stack of unhandled violation exceptions on Kernel white initalizing/loading D3D library/device. In other words - the game crashes, before even executing the main data, which makes it impossible to track anything in the memory, as nothing is loaded there (except for Microsoft/Piniata Paradise bitmaps). How I'd love to get my hands on official Xbox devkit/libraries to more deeply check it out.

From the BanjoXD file:
Section "text" (offset 0x00002000-0x0052C447) contains 5.16MB of data
"rdata" (0x00749000 - 0x0086288F) contains 1.09MB of data
"data" (0x00863000 - 008DC3CB) contains 485KB of data
"data1" (0x008FC000 - 0x008FC0CF) contains only 208 byes.

The rest are libraries and their headers.
